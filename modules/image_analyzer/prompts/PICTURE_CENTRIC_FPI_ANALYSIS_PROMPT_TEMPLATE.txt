## TEMPLATE INFORMATION
- **Purpose**: Comprehensive FPI analysis with detailed technical documentation

- **Processing Time**: 10-15 minutes per FPI (varies by complexity)
- **Output Format**: Structured TXT with complete validation checklists

## CORE PRINCIPLE
**PICTURE-FIRST ANALYSIS**: Only analyze requirements that have associated images. Exclude all text-only requirements from the report.

## EXECUTION METHODOLOGY

### 1. FPI Discovery and Image Inventory
- Locate target FPI directory and identify all images
- Parse FPI TXT file to find requirements with image references (see: imageX.png)
- **CRITICAL**: Exclude any requirement without explicit image reference
- Create primary focus on visual content analysis

### 2. Picture-Centric Organization Structure
```
=== PICTURE-CENTRIC FPI ANALYSIS REPORT ===
├─ FPI Overview (basic metadata only)
├─ IMAGE INVENTORY & ANALYSIS (primary section)
├─ Image-to-Requirement Cross-Reference
├─ Visual Content Extraction (tables/data from images)
├─ CSV Format Ready Data
├─ Visual Design Technical Analysis
├─ Enhancement Details
├─ Processing Summary
└─ Validation Checklist
```

### 3. Image-First Processing Pipeline
**For Each Image:**
- **Image Analysis**: Content type, dimensions, enhancement details
- **Linked Requirements**: Only requirements that reference this specific image
- **Technical Extraction**: Tables, design specs, color analysis, state matrices
- **State Event Matrix Processing**: For diagrams with input/output signal mappings
- **Quality Assessment**: OCR confidence, conversion readiness
- **Ferrari Compliance**: Telltale codes, design standards

### 4. Requirement Filtering Rules
**INCLUDE ONLY:**
- Requirements with explicit image references: "(see: imageX.png)"
- Requirements that contain "Visual Elements: imageX.png" in source data
- Informational requirements with direct image links

**EXCLUDE COMPLETELY:**
- Text-only requirements without image references
- Behavioral requirements without visual elements
- Audio-only requirements (chimes, buzzers without visual component)
- Pure CAN signal requirements without visual mapping

### Section 4: Visual Content Extraction
```
### DIAGRAM TYPE IDENTIFICATION

**Step 1: Analyze Diagram Structure**
- Identify start/end points (terminators - ovals)
- Count total blocks/shapes and identify types
- Determine flow direction (top-to-bottom, left-to-right)
- Look for decision diamonds vs. process rectangles

**Step 2: Classify Diagram Type**
```
IF (start/end terminators AND sequential blocks) THEN
    → Sequential Flow Diagram (use Block-by-Block Analysis)
ELSE IF (grid pattern AND input/output headers) THEN
    → State Matrix Diagram (use Matrix Extraction)
ELSE IF (multiple decision diamonds AND loops) THEN
    → Complex State Machine (use Hybrid Analysis)
ELSE
    → Mixed Analysis Required
```

### SEQUENTIAL FLOW DIAGRAM ANALYSIS

**Block-by-Block Analysis Process:**
```
FOR each block in diagram:
    1. Identify block shape using FLOWCHART_SYMBOL_REFERENCE_GUIDE.txt:
       - Terminator (oval): Start/End points
       - Process (rectangle): Actions/Operations  
       - Decision (diamond): Conditional logic
       - Delay (curved rectangle): Wait periods
       - Connector (circle): Flow continuation
    
    2. Extract block content:
       - Action description
       - Preconditions (what must be true before this block)
       - Decision criteria (if decision diamond)
       - Time requirements (if delay block)
       
    3. Map relationships:
       - Input from previous block
       - Output to next block(s)
       - Alternative paths (for decisions)
       - Cyclic returns (block N → block M where M < N)
```

**Decision Point Analysis:**
```
FOR each decision diamond:
    1. Identify decision criteria from block text
    2. Map logical operators (AND, OR, NOT)
    3. Trace YES/NO paths clearly
    4. Document path conditions:
       - Path A: Condition set 1 (typically YES path)
       - Path B: Condition set 2 (typically NO path)  
       - Logic: IF (A AND B) OR (C) THEN next_block
    5. Check for error handling paths
```

**Sequential Flow State Matrix:**
```
| Block | Preconditions | Action | Conditions Checked | Output State | Next Block |
|-------|---------------|--------|--------------------|--------------|------------|
| 1     | Initial_State | Action_Description | N/A | Resulting_State | Block_2 |
| 2     | Block_1_Complete | Action_Description | Decision_Logic | Decision_State | Block_3/Block_X |
| N     | Block_N-1_State | Final_Action | Termination_Check | Final_State | Terminator |
```

**Cyclic Flow Pattern Recognition:**
```
1. Map all flow paths from start to end
2. Identify blocks that reference previous blocks (Block N → Block M where M < N)
3. Document cycle conditions:
   - Entry point: Block where cycle begins
   - Exit conditions: When cycle terminates  
   - Loop logic: Conditions for continuation vs. exit
```

### STATE EVENT MATRIX EXTRACTION

**For Traditional State Matrices:**
- **Input Parameters:** Clearly define input signals and their possible values
- **Initial State:** Explicitly document the initial state of the system 
- **Output Parameters:** Define output signals and their possible values
- **Complete Combinations:** Ensure all possible combinations included

**Matrix Format:**
| Input_1 | Input_2 | Initial_State | Output_1 | Output_2 | State Description |
|---------|---------|---------------|----------|----------|-------------------|
| Value_1 | Value_2 | Initial_Value | Value_A  | Value_B  | [State_Name]      |
| Value_X | Value_Y | Initial_Value | Value_X  | Value_Y  | [State_Name]      |

### CAN SIGNAL INTERPRETATION

**Automotive Signal Format:** DOMAIN.SIGNAL_NAME.Parameter
**Examples:**
- HMI_CAN.STATUS_DOORS.Interior_Lock
- HMI CAN.KFHUB1.HMiPoweronkeg  
- INFO CAN.WAKE C NBC.Mainwakests BEM

**Signal Values:** Typically binary (Active/Inactive, ON/OFF, 0/1, Present/Absent)

**Documentation Requirements:**
- Map all CAN signals with full hierarchical names
- Define all possible values for each signal
- Document signal dependencies and timing requirements
```

## IMPORTANT NOTE ON FILE HANDLING

The analysis scripts (`run_srs_analysis.sh` and `run_srs_analysis_single.sh`) have been updated to check if an analysis file already exists for a feature before creating a new placeholder file. This prevents accidentally overwriting existing analyses. The scripts now:

1. Check if an analysis file already exists for the feature
2. If an analysis file exists, skip the placeholder creation
3. If no analysis file exists, create a new placeholder file with the current date in the filename

When running the scripts, you will see a message indicating whether an analysis file already exists for each feature. If it does, the script will skip that feature to avoid overwriting the existing analysis.


## 1. OVERALL OBJECTIVE

Your primary task is to create a comprehensive and methodical test management infrastructure for validating Software Requirements Specifications (SRS) for an in-vehicle infotainment system. This involves two main phases for each SRS feature:

1.  **Detailed Analysis**: Systematically analyze the provided SRS feature document (`.txt` file) by yourself with your own AI capabilities in a comprehensive, robust, methodical and detailed method
2.  **Test Case Generation**: Create detailed, methodical, efficient, and traceable test cases based on the analysis that should have a technical language and logic by yourself with your own AI capabilities in a comprehensive, robust, methodical and detailed method
3.  **Parse the data generated at point 1 and point 2 using a .PY script to be able to run it on multiple files in a row
The final output for each feature will be a single, consolidated analysis and test case document.

## 2. PHASE 1: DETAILED ANALYSIS

For each SRS Feature, create a robust and detailed analysis document. The analysis must be structured with the following sections.

### 2.1. Req Feature Overview and Approval Status

-   **Feature ID and Name**: Extract from the source document.
-   **Brief Description**: Write a concise summary of the feature's purpose and functionality. This should clearly explain what the feature does, its main purpose, and how it contributes to the overall system.
-   **Responsible Domain**: Extract from the source document.
-   **Test Stage**: Extract from the source document.
-   **Approval Status**: Check the "Grooming Status" field. If it is not "Approved", mark the feature as "NOT APPROVED" and **stop all further analysis and test case creation for this feature.**
-   **Requirements Approval**: Include a count of approved requirements vs. total requirements (e.g., "15/17 requirements approved (2 obsolete)")
-   **Analysis Date**: Include the current date in YYYY-MM-DD format.

### 2.2. Requirements Summary

-   Summarize all key functional requirements from the source document in a methodical manner using technical approaches
-   Clearly distinguish between active requirements, informational items, Referenced Requirements and exclude the obsolete requirements.

#### 2.2.1 CRITICAL REQUIREMENT ANALYSIS FORMAT (MANDATORY COMPLIANCE)

**ABSOLUTE REQUIREMENT**: You __must__ create **INDIVIDUAL ENTRIES** for each requirement ID. **NO GROUPING OR COMBINING ALLOWED**.

**MANDATORY FORMAT FOR EACH REQUIREMENT**:
```
**Req. ID**: [EXACT_REQUIREMENT_ID] - [SPECIFIC_REQUIREMENT_TITLE]
- **Testability**: [High/Medium/Low] - [SPECIFIC_VERIFICATION_METHOD]
- **Dependencies**: [LIST_SPECIFIC_DEPENDENCIES]
- **Implementation**: [SPECIFIC_IMPLEMENTATION_DETAILS]
```

**VALIDATION CHECKPOINTS**:
- [ ] **Individual Entry Check**: Each requirement ID has its own separate entry
- [ ] **No Combined Entries**: No entries like "3828029, 3793745, 3793757 - Combined Description"
- [ ] **Specific Titles**: Each entry has a specific, unique title describing that requirement only
- [ ] **Complete Information**: Each entry includes testability, dependencies, and implementation
- [ ] **Source Verification**: Each requirement ID matches exactly with the source SRS document

**FORBIDDEN PRACTICES**:
- ❌ **Combined requirement entries** (e.g., "3828029, 3793745 - Area 1 Specifications")
- ❌ **Generic descriptions** that could apply to multiple requirements
- ❌ **Placeholder requirement IDs** not found in the source document
- ❌ **Grouped functional areas** instead of individual requirement analysis

-   For each requirement, you __must__ include:
    * Testability assessment (how the requirement can be verified)
    * Dependencies on other requirements or signals
    * Implementation considerations or challenges
    * Note any obsolete or problematic requirements with clear marking

### 2.2.1 Requirement Quality Assessment

-   For each requirement with RQA issues:
    * Analyze the specific issues identified (e.g., compound requirements, missing units)
    * Create a mitigation strategy for each issue type:
        - For compound requirements: Break down into atomic test steps
        - For missing units: Define appropriate units and tolerances for testing
        - For negative statements: Convert to positive verification criteria
        - For unclear terms: Define specific, measurable criteria
    * Flag requirements with low RQA scores (<70) for additional test coverage
    * Document how the test cases will address these quality issues

### 2.3. Advanced Visual Elements Analysis

#### 2.3.1 CLIP-Based Image Classification Integration (MANDATORY WORKFLOW)

-   You __must__ use the existing **CLIP classification system** as the **mandatory first step** for each image in the feature folder:
    * **Step 1 - Retrieve CLIP Results**: For each image in the feature folder, look up its classification in `merged_clip_predictions.csv`
    * **Step 2 - Extract Classification Data**: Obtain the predicted category, confidence level, and any additional metadata
    * **Step 3 - Validate Classification**: Ensure the CLIP prediction aligns with the image content and context within the SRS feature
    * **Step 4 - Handle Low Confidence**: For images with confidence < 0.7, consider secondary category analysis or manual verification
    * **CRITICAL REQUIREMENT**: **NO IMAGE ANALYSIS** may proceed without first completing the CLIP classification lookup

-   The **13 specialized automotive categories** used by the CLIP system are:

    1. **HMI DISPLAY LAYOUTS**: Full instrument-cluster screens showing multiple UI regions together (speed/RPM/fuel/warnings). A single cohesive dashboard display, NOT a lone warning icon and NOT a blocky flowchart of boxes and arrows.

    2. **CONFIGURATION TABLES**: Technical tables with rows and columns whose cells contain only text or numbers. May include borders, headers, or colored/shaded text BUT NOT COLORED ICONS but NO pictogram shapes or dashboard symbols. No icon silhouettes in any cell.

    3. **TELLTALE ICONS & INDICATORS**: Single isolated car warning icon or symbol on a plain background with no other dashboard elements. Only one warning light (battery, oil, temperature, etc.).

    4. **TABLE WITH TELLTALES**: Tables that contain automotive warning pictograms INSIDE their cells. At least one cell shows a recognizable colored image with an icon (battery, oil lamp, ABS, seatbelt), i.e., a drawn symbol with an outline. Not just colored or shaded cells—must be a visible pictogram.

    5. **SYSTEM ARCHITECTURE DIAGRAMS**: Block diagrams of modules (ECUs, sensors, controllers) as rectangular boxes connected by lines or arrows. Focus on components and data or wiring connections; no logic gate symbols.

    6. **PROCESS FLOW DIAGRAMS**: Process flow diagrams showing sequential steps and decision logic with connected shapes. Contains rectangular process boxes, diamond decisions with Yes/No branches, start/end terminators, and arrows for workflow. Must not contain a horizontal time axis with stacked HIGH/LOW waveform rows.

    7. **STATE FLOW DIAGRAMS**: State machine diagrams with circular/oval states connected by arrows and labeled transitions. Emphasis on state bubbles and directional connectors. NOT a waveform with a horizontal time axis.

    8. **DIGITAL TIMING DIAGRAMS**: Digital timing diagrams with a horizontal time axis, stacked HIGH/LOW waveform rows, rectangular step transitions, and vertical alignment guides. Shows signal states over time, with synchronized markers and temporal relationships. Not a flowchart or wiring schematic.

    9. **ANALOG/THRESHOLD TIMING DIAGRAMS**: Plots versus time or parameter with axes and units. Single or few continuous traces (ramps, slopes) and threshold markers, or ON/OFF versus a non‑time variable such as speed or temperature.

    10. **COLOR AND GRADIENTS**: Color charts, gradient scales, or color calibration patterns. No text or icons.

    11. **TECHNICAL SPECIFICATIONS**: Technical specification documents or parameter lists that don't fit other categories but contain important technical information.

    12. **MULTI-CONDITION LOGIC**: Complex logic diagrams showing multiple conditions, decision trees, or multi-variable relationships that don't fit standard flowchart patterns.

    13. **GENERAL TECHNICAL DIAGRAMS**: Any other technical diagrams that don't fit the above categories but contain important automotive system information.
